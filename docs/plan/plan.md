# Vapor Only Runtime 微阶段总计划

本文档不再直接给 AI 一整份大计划，而是把实现拆成“总索引 + 阶段总览 + 微阶段任务”。

## 为什么要这样拆

之前的 1.md、2.md、3.md 仍然偏粗，容易出现两个问题：

1. 单次上下文过长，AI 需要同时记住太多模块和约束。
2. 一次改动跨越太多文件，容易把不相关的问题一起带出来。

现在的用法改为：

1. 先读本文件。
2. 再只读当前阶段总览。
3. 最后只读当前一个微阶段文件。

## 读取规则

每次实现时，最多只加载以下 3 个文件：

1. 本文件
2. 当前阶段总览
3. 当前微阶段任务

不要一次把 01.md 到 13.md 全部放进上下文。

## 执行规则

每个微阶段都应满足以下约束：

1. 目标只解决一个清晰问题。
2. 最好只跨 1 到 2 个包。
3. 最好只改 2 到 6 个核心文件。
4. 能单独提交、单独验证、单独回退。
5. 如果实现过程中发现文件面继续膨胀，应先回到计划文档再次拆分。

## 推荐执行顺序

建议严格按编号顺序推进，不要跳步：

1. 01 到 05 解决协议、入口和 compat 基础。
2. 06 到 10 解决编译器、路由和内置组件主路径。
3. 11 到 13 解决对外类型、compat 收缩和文档发布。

## 验证层级

### 微阶段完成后

至少执行该微阶段文档里要求的最小验证。

### 一个阶段完成后

统一执行：

1. pnpm check
2. pnpm test-unit
3. pnpm app-build
4. cd packages/runtime-vapor && npm run build-dev
5. cd packages/swc-plugin-rue && npm run check && npm run test

## 最终目标

当下面 5 条同时成立时，才算默认路径真正去掉虚拟 DOM：

1. 编译后的应用主路径不再依赖 h、createElement、patch 和 render vnode。
2. Router、Teleport、Transition、TransitionGroup、useComponent 默认实现不再构造 vnodeLike。
3. 生命周期与 effect scope 清理不再依赖 VNode before_unmount。
4. VNode 只存在于 compat 子路径，或者已在主包中完全删除。
5. 文档已经从“虚拟 DOM 是默认机制”改写为“Block / Vapor 是默认机制”。
