# 清单 2：补齐 runtime custom-elements 契约，让编译产物可以稳定依赖

现在的 `useCustomElement()` 已经足以支撑手写 demo，但编译器一旦开始批量产出 custom element 构造器，就不能再把 runtime 看成“示例级能力”。编译产物需要一套稳定、明确、可文档化的宿主协议，包括 props、attribute、事件、slot、样式、宿主生命周期和豁免语义。

## 目标

1. 冻结 custom-elements runtime 的最小稳定契约，让编译产物知道该依赖什么。
2. 把当前 demo 可用但边界模糊的行为，收口成明确的规则或显式不支持项。
3. 为后续 swc 产物提供统一的注册、样式、事件和 slot 行为基础。

## 当前阻塞

1. `packages/runtime/src/custom-elements.ts` 目前更偏“手工包装器”视角，还没有编译器消费的稳定协议描述。
2. `this.$host` 尚未实现，`props.children` 与原生 slot 分发并不是同一协议。
3. 自定义元素专用的 provide/inject、作用域 slot、样式抽取与注入还没有统一答案。
4. 当前事件桥接虽已工作，但事件命名、`detail` 结构、宿主监听约定还缺编译器级文档冻结。
5. 还没有明确规定编译器应当把复杂值落到宿主属性、`el.props`，还是其他桥接层。

## 执行清单

- [ ] 决定编译器产物对 runtime 的依赖面：最小要求是否仅限 `useCustomElement`，还是需要额外的 registry/helper。
- [ ] 冻结 props 协议：
  - 字符串和标量优先走 attribute；
  - 复杂值走宿主属性或 `el.props`；
  - 删除属性时如何同步为 `undefined`；
  - 组件子树如何感知这些变化。
- [ ] 冻结事件协议：`emitted(props)` 触发的宿主 `CustomEvent` 命名、`detail` 数组结构、`bubbles/composed` 的固定值。
- [ ] 冻结 slot 协议：默认 slot、命名 slot、light DOM 模式下的限制、`props.children` 的不兼容边界。
- [ ] 明确 styles 协议：是继续要求编译器显式传 `styles: string[]`，还是需要后续接样式收集链路。
- [ ] 明确哪些特性在 native custom element 模式下一期不支持，例如 `this.$host`、作用域 slot、custom-element 专用 provide/inject。
- [ ] 为编译器将来依赖的行为补 focused regression，避免 swc 接入后 runtime 再改口。

## 阶段边界

1. 这一刀优先冻结 runtime 契约，不直接实现 swc 的 AST 代码生成。
2. 不把“当前手写 demo 能跑”视为完成；必须让编译器产物也能稳定依赖。
3. 不在这一阶段承诺所有 Vue/Rue 组件语义都可映射为 native custom element；不支持项要明确写出。

## 完成定义

1. `useCustomElement()` 的 props、events、slot、styles、hooks 契约已经固定下来。
2. 编译器产物不再依赖未定义行为或“手工 demo 约定”。
3. runtime 文档与回归测试已经能覆盖编译模式需要依赖的宿主协议。
