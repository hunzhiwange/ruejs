# 清单 3：实现定义侧编译，把函数组件声明包装成 custom element 构造器

定义侧包装是 compile-to-custom-element 的最小闭环。只要能把一个 opt-in 的函数组件稳定编译为 `useCustomElement(Foo, options)` 形式的构造器导出，就能先证明编译器与 runtime 的连接点是可行的。这个阶段先不碰消费侧 `<Foo />` 调用协议，避免一开始把问题面摊得过大。

## 目标

1. 在 swc 插件里新增 module-level pass，把候选函数组件包装成 custom element 构造器。
2. 保持现有 pre / vapor 逻辑不被破坏，让组件本体仍按原有路径先被编译成 Rue 组件。
3. 支持稳定的导出形态与可选自动注册逻辑。

## 当前阻塞

1. `packages/swc-plugin-rue/src/lib.rs` 目前只有 pre transform 与 vapor transform，没有定义侧包装 pass。
2. `packages/swc-plugin-rue/src/imports.rs` 的 helper 注入集合里没有 `useCustomElement`。
3. 当前没有“哪个声明是 opt-in 组件、哪个导出需要变成 custom element 构造器”的模块级收集逻辑。
4. 当前没有约定：包装后是保留原组件标识符，还是完全替换成构造器导出。
5. 当前没有对自动 `customElements.define()` 的 AST 生成、去重和测试覆盖。

## 重点文件

1. `packages/swc-plugin-rue/src/lib.rs`
2. `packages/swc-plugin-rue/src/imports.rs`
3. `packages/swc-plugin-rue/src/pre/transform.rs`
4. `packages/swc-plugin-rue/src/pre/helpers.rs`
5. `packages/swc-plugin-rue/src/vapor/visitor.rs`
6. 新增的 custom-element 包装 pass 文件

## 执行清单

- [ ] 在 `lib.rs` 新增 compile-to-custom-element 配置解析，并把配置传给新的 module-level pass。
- [ ] 复用 pre 阶段已有的函数组件识别逻辑，确定哪些声明是可包装的目标。
- [ ] 新增定义侧包装 pass，在组件编译完成后生成类似 `const FooElement = useCustomElement(Foo, options)` 的 AST。
- [ ] 决定导出策略：
  - 保留 `Foo`，追加 `FooElement`；
  - `export default FooElement`；
  - 或保留两者并允许配置控制。
- [ ] 如配置要求自动注册，则生成带 `customElements.get(tag)` 守卫的注册代码。
- [ ] 在 `imports.rs` 里补 `useCustomElement` 的按需导入与稳定排序。
- [ ] 为定义侧包装新增 swc fixture / snapshot tests，覆盖函数声明、箭头函数、default export 和 named export。

## 阶段边界

1. 这一刀只打通“定义侧变成构造器”，不把 `<Foo />` 调用点同时改成原生标签。
2. 不允许为省事直接替换原组件标识符，导致同模块内自引用或调试命名丢失；优先保留原组件，再额外生成构造器值。
3. 不在这里解决跨模块 tag 分发与 slot 降级，那属于 [4.md](./4.md) 的范围。

## 完成定义

1. opt-in 的函数组件已经可以由 swc 直接编译出 custom element 构造器导出。
2. `useCustomElement` 导入由编译器自动注入，不需要用户手写补 import。
3. 现有 vapor 编译结果没有被定义侧包装破坏，未 opt-in 的组件保持原行为。
