# 清单 1：定义 compile-to-custom-element 模式、tag 命名与注册协议

当前最大的缺口不是 runtime 能不能包一层 `useCustomElement()`，而是编译器根本不知道什么时候该进入 custom element 模式、该生成什么 tag、是否需要自动注册、以及哪些组件不应该被纳入这套协议。如果这些前置约束没先定下来，后面的 runtime 和 swc 改造只会反复返工。

## 目标

1. 给 `packages/swc-plugin-rue` 增加稳定的 compile-to-custom-element 配置入口，而不是继续只有 `{ vapor: boolean }`。
2. 定义组件到 custom element tag 的命名策略、冲突策略与前缀策略。
3. 定义构造器导出、自动 `customElements.define()`、以及 manifest/注册表三种注册模式的边界。
4. 明确哪些框架控制组件属于白名单豁免，不能直接纳入“全部原生组件改造”。

## 当前阻塞

1. `packages/swc-plugin-rue/src/lib.rs` 目前只读取 `vapor` 布尔值，没有 compile-to-custom-element 的配置入口。
2. 仓库里没有 `Ident -> tagName` 的统一生成策略，也没有冲突检测规则。
3. 没有统一约定导出形态：是 `export default FooElement`、`export const FooElement`，还是保留原 `Foo` 再追加构造器。
4. 没有统一约定注册时机：编译期自动注册、运行期 bootstrap 注册、还是只导出构造器交给调用方定义。
5. 目前没有明确列出 `Teleport`、`Transition`、路由控制组件等“不可默认 native 化”的豁免名单。

## 执行清单

- [ ] 设计 compile-to-custom-element 配置结构，例如：`{ enabled, mode, tagPrefix, shadowRoot, autoDefine, export, include, exclude }`。
- [ ] 决定 opt-in 入口：全文件配置、文件级 pragma、导出级注解，还是多者并存。
- [ ] 设计 tag 命名规则：`FooBar -> foo-bar` 是否需要仓库级前缀，如何处理同名冲突与跨目录重复。
- [ ] 设计注册策略：
  - 只导出构造器；
  - 构造器 + `register()` 辅助；
  - 自动 `customElements.define()`；
  - manifest 驱动的外部统一注册。
- [ ] 明确 dev / HMR 场景下的重复注册策略，避免直接在运行时抛 `DOMException`。
- [ ] 输出一份框架控制组件的豁免名单和原因说明，作为后续 [5.md](./5.md) 的输入。
- [ ] 为配置解析和 tag 命名补最小 swc fixture / snapshot 测试。

## 阶段边界

1. 这一刀只定义协议与配置，不直接实现 AST 包装与调用侧降级。
2. 不在没有豁免名单时就宣称“所有组件都可 native 化”；框架控制组件必须先从范围里剥离。
3. 不允许把 tag 命名策略留给人工约定；编译模式必须有稳定、可验证的规则。

## 完成定义

1. 编译模式配置结构已经定稿，并写回计划或文档。
2. tag 命名与注册策略已定稿，重复注册/HMR 行为已有明确策略。
3. 框架控制组件豁免名单已存在，后续阶段不再对其“默认 native 化”做隐含假设。
