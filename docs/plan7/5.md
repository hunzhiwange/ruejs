# 清单 5：完成仓库组件迁移盘点、白名单豁免与 app/examples 落地

“全部原生组件改造”不能只停留在编译器层。仓库本身必须回答三个问题：哪些组件已经能安全 native 化，哪些组件属于框架控制白名单需要继续保留 Rue 原协议，哪些业务组件与示例页面应该作为第一批迁移样板。如果这一步不做，最终只会得到一个理论上可用、仓库里却没人真正用的模式。

## 目标

1. 给仓库里的组件做迁移盘点，区分业务组件、普通函数组件、框架控制组件和显式豁免项。
2. 选出第一批迁移对象，把 app/examples/docs 里的真实页面切到 native custom element 模式。
3. 建立“已迁移 / 暂缓 / 豁免”的明确清单，避免后续范围失控。

## 当前阻塞

1. 仓库里还没有一份组件分类清单，不知道哪些组件天然适合 native 化。
2. `Teleport`、`Transition`、路由控制组件等仍有明显框架控制语义，不能被简单地视为普通 custom element。
3. app/pages 与 docs/examples 目前只有手写 custom elements 工作台，没有真实的业务组件迁移样板。
4. 没有仓库级 tag 命名和注册规则时，示例页与真实页面无法大规模落地。

## 重点文件

1. `app/pages/examples/WebComponents.tsx`
2. `app/pages/examples/*`
3. `app/pages/site/*`
4. `app/router/index.ts`
5. `packages/runtime/src/components/*`
6. `packages/router/*`
7. `docs/api/custom-elements.md`
8. `docs/guide/extras/web-components.md`

## 执行清单

- [ ] 先盘点仓库组件，把它们分成：
  - 业务/展示组件，可直接 native 化；
  - 依赖父子控制语义的框架控制组件，需要白名单保留；
  - 需要额外适配后才能 native 化的中间类。
- [ ] 给第一批迁移对象定范围，优先选择 examples、站点侧普通展示组件和简单交互组件。
- [ ] 在 app/examples 里补至少一组“编译产物驱动”的 native custom element demo，而不只是手写 `useCustomElement()`。
- [ ] 在站点文档里增加“哪些组件已支持 native 化、哪些仍豁免”的公开口径。
- [ ] 对框架控制组件给出明确结论：继续保留 Rue 协议、做专用编译策略，还是彻底不纳入本阶段。
- [ ] 维护一份迁移台账，列清每个组件的状态与剩余问题。

## 阶段边界

1. 这一刀不追求一次性把仓库所有组件都切完；目标是建立清单、白名单和第一批样板迁移。
2. 不为了凑“全部”而把明显依赖框架控制语义的组件强行 native 化。
3. 不允许只有编译器实现、没有真实页面迁移样板；仓库里至少要有可运行的落地页。

## 完成定义

1. 仓库组件已经有“已迁移 / 暂缓 / 豁免”的明确台账。
2. app/examples 或站点页面已经有真实 native custom element 编译样板。
3. 框架控制组件的处理策略已经文档化，不再处于模糊状态。

## 推荐验证

1. `pnpm app-build`
2. `pnpm vitest run app/**`
3. 浏览器手工 smoke test：至少确认一个真实页面里的 native custom element 组件链路正常
