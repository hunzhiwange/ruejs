# 清单 6：完成类型、HMR、测试矩阵、文档与发布收口

前面几刀完成后，仓库里会第一次同时存在“Rue 组件协议”和“native custom element 协议”两条编译与消费路径。如果没有类型、HMR、测试与发布口径的收口，最终很容易出现：源码能编、demo 能跑、但编辑器类型错乱、热更新炸注册、CI 没覆盖、发布说明也说不清楚。

## 目标

1. 让 native custom element 编译模式具备基础类型体验与 HMR 安全策略。
2. 建立从 swc fixture 到 runtime jsdom 再到 app smoke test 的完整验证矩阵。
3. 完成文档、迁移说明与发布口径，避免只在内部记忆里存在实现细节。

## 当前阻塞

1. 当前没有生成 custom tag 对应类型的机制，编辑器层仍缺“组件名 -> 原生标签 props”这类补充信息。
2. 当前没有统一的 `customElements.get(tag)` / HMR 重注册策略规范。
3. 当前没有从 TSX 源码经 swc 编译、再到浏览器运行 native custom elements 的端到端验证链路。
4. 当前文档虽然有 runtime custom elements API，但没有“仓库 compile-to-custom-element 模式”的完整迁移指南。

## 重点文件

1. `packages/rue/jsx.d.ts`
2. `packages/rue/env.d.ts`
3. `packages/swc-plugin-rue/tests/*`
4. `packages/runtime/__tests__/custom-elements.spec.tsx`
5. `app/pages/examples/WebComponents.tsx`
6. `docs/api/custom-elements.md`
7. `docs/guide/extras/web-components.md`
8. `CHANGELOG.md`

## 执行清单

- [ ] 设计并实现 native custom element 编译模式的类型补充方案：tag manifest、声明生成、或更窄的 JSX augment 方案。
- [ ] 定义 HMR 与重复注册策略，统一使用 `customElements.get(tag)` 守卫或等价机制。
- [ ] 建立测试矩阵：
  - swc fixture / snapshot tests；
  - runtime jsdom tests；
  - app/examples smoke tests；
  - 必要时补浏览器端 e2e。
- [ ] 为 compile-to-custom-element 模式写迁移指南，说明适用范围、限制项、slot 与 props 语义差异。
- [ ] 在发布说明里明确这是默认行为、opt-in 行为，还是实验性行为，以及 semver 影响。

## 阶段边界

1. 不把“类型先缺着、以后再补”当作可发布状态；编辑器层编译模式本身就要求类型体验至少不倒退。
2. 不接受只有 swc snapshot、没有 runtime/app 级验证的测试矩阵。
3. 不在文档没有写清限制项时就默认推广到仓库全量组件。

## 完成定义

1. native custom element 编译模式具备基础类型与 HMR 安全策略。
2. 从编译器到 runtime 再到 app 的验证链路已经建立。
3. 文档、迁移指南和发布说明已经能准确表达这个模式的能力与限制。

## 推荐验证

1. `cargo test -p swc_plugin_rue`
2. `pnpm vitest run packages/runtime/__tests__/custom-elements.spec.tsx packages/runtime/__tests__/rue.event.spec.tsx packages/runtime/__tests__/compat-entry.spec.ts`
3. `pnpm check`
4. `pnpm app-build`
5. 浏览器 HMR smoke test
